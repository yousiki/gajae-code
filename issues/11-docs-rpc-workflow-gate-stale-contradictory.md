# 11 — `docs/rpc.md` workflow-gate sub-protocol is internally contradictory and stale

- **Severity:** Low (documentation correctness)
- **Scope:** `docs/rpc.md:383-447` (first "Workflow Gate Sub-Protocol" section) vs `docs/rpc.md:724-758` (later section) vs source-of-truth `packages/coding-agent/src/modes/rpc/rpc-types.ts:457-470` (`RpcWorkflowGate`)
- **Surface:** RPC protocol documentation

## Summary

`docs/rpc.md` documents the `workflow_gate` frame **twice, inconsistently**, and
the first version contradicts the actual `RpcWorkflowGate` type.

Source of truth (`rpc-types.ts`):

```ts
export interface RpcWorkflowGate {
  type: "workflow_gate";
  gate_id: string;            // "wg_<run>_<stage>_000001"
  stage: RpcWorkflowStage; kind: RpcWorkflowGateKind;
  schema: RpcJsonSchema; schema_hash: string;
  options?: RpcWorkflowGateOption[];     // { value, label, description? }
  context: RpcWorkflowGateContext;       // { title?, prompt?, summary?, stage_state?, artifact_refs?, language? }
  created_at: string; required: true;
}
```

First doc section (lines ~387-407) disagrees:

```json
{ "type": "workflow_gate", "gate_id": "123", "stage": "ralplan:approval",
  "kind": "approval", "schema": { "type": "boolean" },
  "options": ["Approve", "Reject"],
  "context": { "method": "confirm", "skill": "ralplan", "phase": "approval" } }
```

- `options` is a **string array** here vs `RpcWorkflowGateOption[]` (objects).
- `context` uses `{ method, skill, phase }` vs the real `{ title, prompt,
  summary, … }`.
- Missing `schema_hash`, `created_at`, `required`.
- `gate_id: "123"` and the prose claims it "matches the paired interactive UI
  request id and is monotonic because it is generated by the same process-wide
  `Snowflake` id source" — but the real `gate_id` is broker-generated
  `wg_<run>_<stage>_NNNNNN` (also stated correctly at line 748).
- `stage` is shown as `"ralplan:approval"` but the type is `RpcWorkflowStage`
  (`"deep-interview" | "ralplan" | "ultragoal"`).

The later section (724-758) matches the type. So the document contradicts both
the code and itself.

## Impact

- A host implementer following the first section builds the wrong gate parser
  (string options, wrong context keys, wrong gate_id assumptions).
- Error-code naming also drifts: the first section names
  `invalid_workflow_stage` while construction errors elsewhere use
  `invalid_workflow_gate_schema`.

## Suggested fix

Delete/replace the stale first "Workflow Gate Sub-Protocol" block (383-447) so a
single description matches `RpcWorkflowGate` / `RpcWorkflowGateOption` /
`RpcWorkflowGateContext` and the later section. Generate the example from the
types if possible to prevent re-drift.
