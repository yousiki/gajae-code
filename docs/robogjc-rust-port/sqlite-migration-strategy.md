# robogjc SQLite migration strategy

## Python-era schema baseline

`python/robogjc/src/db.py` is the Phase 2 compatibility contract. The Rust port in `crates/robogjc/src/db.rs` creates the same table and index inventory:

- `events`
- `issues`
- `tool_calls`
- `submissions`
- `pending_closures`
- indexes: `events_state_received`, `tool_calls_issue`, `submissions_login_ts`, `pending_closures_state_close_at`

SQLite connection setup remains WAL-compatible with the Python implementation:

```sql
PRAGMA journal_mode = WAL;
PRAGMA synchronous = NORMAL;
PRAGMA foreign_keys = ON;
```

Atomic claim/admission paths use `BEGIN IMMEDIATE` semantics in both implementations for event dequeue, submission admission, and pending-closure claims.

## `PRAGMA user_version` scheme

Rust migrations use `PRAGMA user_version` as the canonical schema version.

| Version | Meaning | Compatibility |
| --- | --- | --- |
| `0` | Unstamped Python-era database. Python never set `user_version`. | Must open in Python and Rust. |
| `1` | Python-era DDL stamped by Rust. This is a no-op-compatible stamp, not a shape change. | Must open in Python and Rust. |

### Migration `0 -> 1`

`0 -> 1` is intentionally conservative:

1. Run the Python-era `CREATE TABLE IF NOT EXISTS` / `CREATE INDEX IF NOT EXISTS` schema.
2. Apply idempotent legacy column repairs already present in Python (`issues.classification`, `events.model`) when opening older ad-hoc databases.
3. Set `PRAGMA user_version = 1` only after the schema is present.
4. Preserve every existing row and column value.

Future migrations must add a new row to the table above and must keep rollback compatibility explicit. While Python rollback remains possible, additive columns are allowed only when Python safely ignores them; destructive changes are prohibited.

## Fixture generation

The compatibility fixture is committed at:

```text
artifacts/robogjc/db/python-era-v1.sqlite
```

It is generated by the Python implementation, not by handwritten SQL:

```bash
/tmp/robogjc-uv/bin/python crates/robogjc/tests/fixtures/phase2/seed_python_era.py artifacts/robogjc/db/python-era-v1.sqlite
```

The fixture intentionally keeps `PRAGMA user_version = 0` until Rust opens it. Seed coverage includes:

- queued/running/done/failed/skipped events, including `model` and skipped error text
- issues with classification, branch, session directory, and PR mapping
- successful and failed `tool_calls`
- submissions for multiple normalized logins
- pending, claimed, closed, and cancelled pending closures

## Compatibility tests

Rust tests cover the live migration and rollback contract:

- `db_migrates_python_era_fixture`: copies the Python-generated fixture, opens it with Rust, asserts seeded rows are intact, and verifies `user_version = 1`. The test fails loudly when the committed fixture is missing.
- `db_python_rust_compatibility`: Rust writes rows into a migrated database, then shells out to `/tmp/robogjc-uv/bin/python` and reads the same rows via `robogjc.db.Database`. This interpreter-dependent check is explicitly ignored by default; run it with `ROBGJC_PY_COMPAT=1 cargo test -p robogjc db_python_rust_compatibility -- --ignored` for the full Python/Rust compatibility gate.

Default verification commands for Phase 2 are:

```bash
cargo test -p robogjc
/tmp/robogjc-uv/bin/python -m pytest -x --import-mode=importlib python/robogjc/tests/test_db.py
```

## Intentional Python deviations

Pending-closure finalization is intentionally stricter in Rust than in `python/robogjc/src/db.py`: Python's `finalize_closure` can overwrite an already-terminal `closed` row with `cancelled` because it updates by `issue_key` without a state guard. Rust requires `state = 'claimed'`, returns whether a row changed, and treats double-finalize, cancel-after-finalize, and requeue-after-finalize as no-ops. This preserves the documented `claimed -> terminal` contract even though it is a bug-for-bug deviation from the Python implementation.

## Operator backup, dry-run, and rollback runbook

Before applying a Rust migration to an operator database:

1. Stop mutating consumers for the affected repo allowlist.
2. Checkpoint WAL state so the main database file contains all committed pages:
   ```sql
   PRAGMA wal_checkpoint(TRUNCATE);
   PRAGMA integrity_check;
   PRAGMA user_version;
   ```
3. Copy the database and sidecar files as a set to a timestamped backup location:
   - `<name>.sqlite`
   - `<name>.sqlite-wal` when present
   - `<name>.sqlite-shm` when present
4. Run the Rust migration against the backup copy first.
5. Validate the dry-run output:
   ```sql
   PRAGMA integrity_check;
   PRAGMA user_version;
   SELECT name, type FROM sqlite_schema
     WHERE type IN ('table','index')
     ORDER BY type, name;
   ```
6. Run a read compatibility smoke with Rust and Python while Python rollback remains in scope.
7. Apply the migration to the live database only after the dry-run passes.

Rollback steps:

1. Stop Rust and any other mutating consumer.
2. Restore the timestamped database backup and sidecar files together; never mix a main DB file with WAL/SHM files from another point in time.
3. Re-run `PRAGMA integrity_check` and record `PRAGMA user_version`.
4. Restart the previous active consumer.
5. Record whether the restored database is unstamped Python-era (`0`) or stamped compatible Python-era (`1`).

## Deletion boundary

The bidirectional requirement remains until the Python implementation is deleted. After deletion, Rust migrations may remove Python-only compatibility code, but only in a migration that records a new `PRAGMA user_version` and documents the irreversible boundary in this file.
