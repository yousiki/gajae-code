## Summary
The Planner plan is directionally sound and matches the deep-interview spec on the big topology: Rust protocol core, pi-natives N-API embedding, codex-compatible JSON-RPC 2.0, namespaced gjc extensions, migration before atomic deletion. I recommend REQUEST CHANGES before execution because the plan under-specifies the hard parts that can invalidate the architecture: current TS AgentSession is not process-global-neutral, the notification precedent is single-session, and Codex interop needs a pinned subset plus schema proof.

Steelman antithesis: keep the current TS AgentSession registry and agent-wire/RPC scheduler as the authoritative runtime, then add a thin codex-shaped TS app-server facade while the Rust core port advances. That would avoid N-API multi-thread callback risk and preserve mature RPC flow control, but it would delay the Rust-owned protocol source of truth and keep future native-core migration coupled to TS-only runtime assumptions.

Synthesis: keep Rust owning wire protocol, connection state, ids, schemas, transport backpressure, and the thread registry, but make execution safety explicit: each Rust ThreadHandle must wrap a TS backend with generation tokens, per-thread serial lanes, bounded TSFN queues, and a documented identity model separating codex `threadId` from mutable GJC `sessionId`.

## Analysis
Evidence inspected:
- Spec `.gjc/_session-019f1c30-3754-7000-b634-75d54f2dc70c/specs/deep-interview-gjc-app-server.md` requires hard replacement, current TS AgentSession bridge, multi-thread `threadId -> AgentSession`, codex core plus gjc extensions, transports, generated TS plus JSON Schema, and deletion of RPC plus notifications SDK.
- Planner `.gjc/_session-019f1c51-6f3a-7000-89c2-5e2fcbb61ee2/plans/ralplan/019f1c51-6f3a-7000-89c2-5e2fcbb61ee2/stage-01-planner.md` covers the right modules and phases, including Rust-owned registry and bounded queues, but leaves several executable contracts implicit.
- Current RPC is single AgentSession per process mode: `packages/coding-agent/src/main.ts` dispatches only `rpc` and `rpc-ui` to `runRpcMode(session)`, and `packages/coding-agent/src/modes/rpc/rpc-mode.ts` subscribes one session through one `AgentWireFrameSequencer` plus one per-session command scheduler.
- Current TS session construction installs process globals: `packages/coding-agent/src/sdk.ts` creates and sets `AsyncJobManager.setInstance`, `MCPManager.setInstance`, active skills/rules, and provider/search preferences for top-level sessions; `AgentSession.dispose` clears owned globals if it owns them.
- Notification Rust precedent is much narrower: `crates/pi-natives/src/notifications.rs` exposes one `NotificationServer` with one config, one handle, and `ThreadsafeFunction` callbacks using `ThreadsafeFunctionCallMode::NonBlocking`; `crates/gjc-notifications/src/server.rs` owns one per-session WS endpoint and action registry.
- Agent events are richer and more timing-sensitive than Codex item notifications: `packages/agent/src/types.ts` has assistant `text_start`, `text_delta`, `thinking_start`, `thinking_delta`, `toolcall_delta`, `done`, and `error`; `packages/coding-agent/src/session/agent-session.ts` emits `message_update` directly to subscribers but queues extension handling, and defers external `agent_end` until prompt in-flight state unwinds.
- Current dependents are live: `python/gjc-rpc/src/gjc_rpc/client.py` builds `gjc --mode rpc`, parses legacy response and event shapes, and exposes many typed methods; `python/robogjc/src/worker.py` relies on that client, `--continue`, host tools, and event listeners.
- Schema CI is TS-origin today: `scripts/generate-json-schemas.ts` emits only config/models schemas from TS/Zod, and workspace Cargo lacks schemars/ts-rs. Codex protocol uses `schemars` plus `ts-rs` and intentionally omits the JSON-RPC `jsonrpc` field in `codex-rs/app-server-protocol/src/rpc.rs`.

Spec compliance: the plan covers the requested replacement and migration scope, but the current plan is not yet concrete enough to guarantee multi-thread safety or Codex-compatible item semantics. Architecture status is therefore WATCH, not CLEAR.

## Root Cause
The fundamental tension is that the desired app-server is a long-lived, multi-thread Rust authority, while the current TS runtime was built around one top-level AgentSession per CLI process with process-wide managers, process-wide protocol handlers, and event timing tailored to RPC, ACP, and notifications. The notifications Rust precedent proves loopback transport and TS callbacks, not concurrent AgentSession ownership.

## Findings
1. HIGH — Multi-thread TS AgentSession isolation is not specified. Evidence: `sdk.ts` installs top-level AsyncJobManager and MCPManager as process globals, while `AgentSession.dispose` may clear them; active skills/rules and provider preferences are also process scoped. Impact: two codex threads in one app-server can clobber internal URL resolution, background jobs, MCP tools, skill state, or disposal. Fix: add a Phase 0 isolation audit and execution contract for all process globals; either create one app-server-wide owner or make them per-session keyed. Add a two-session concurrency test that runs async jobs, MCP/internal URLs, browser/eval ownership, and disposes one thread while the other continues.

2. HIGH — N-API callback flow control is under-specified for many concurrent threads. Evidence: `NotificationServer` pumps a single-session receiver into TSFN calls with `NonBlocking`, while current RPC has an explicit per-session ordered scheduler and fast cancellation lane. Impact: slow JS callbacks or high-volume `message_update` bursts can reorder commands, drop lifecycle events, or accept work that cannot be delivered. Fix: require per-thread serial command lanes plus cancel/read fast lanes, bounded ingress and egress queues, explicit TSFN call failure handling, and overload rejection before accepting `turn/start`. Core lifecycle events must be non-droppable; optional gjc debug/detail events may be droppable only with counters and diagnostics.

3. HIGH — Registry identity ownership needs a precise codex-thread versus GJC-session model. Evidence: AgentSession `newSession`, `switchSession`, and branch/resume operations can change session file and session id while listeners survive; notifications runtime already has special re-key behavior for new sessions. Impact: a Rust `threadId -> AgentSession` map can diverge from TS session identity, stale events can target the wrong thread, and delete/unload can dispose the wrong backend. Fix: make codex `threadId` immutable and Rust-owned; treat mutable GJC `sessionId` and `sessionFile` as backend metadata with generation tokens. Define whether `gjc/session/new` creates a new codex thread or mutates only backend metadata, and reject stale `emitBackendEvent(threadId, generation, event)` callbacks.

4. HIGH — Event normalization must be a state machine, not a table. Evidence: assistant stream events include start/delta/end per content index; `message_update` reaches subscribers ahead of extension ordering; `agent_end` is delayed to avoid false idle; notifications rely on `message_end` before ask `tool_execution_start`. Impact: Codex clients require `item/started` before deltas and exactly one `item/completed`; naive mapping can lose thinking/toolcall details, produce duplicate terminal events, or report idle before prompt finalizers finish. Fix: define and test an event mapper with item accumulators, content-index mapping, per-turn generation, terminal coalescing, and raw `gjc/event` preservation. Cover interleaved text/thinking/toolcall streams, compaction, tool updates, queued steer/follow-up, abort, and redacted thinking.

5. MEDIUM — Codex interop expectations need a pinned v1 subset and stricter command/exec shape. Evidence: Codex app-server README uses omitted `jsonrpc`, `turn/start` input unions, `turn/steer` with `expectedTurnId`, thread statuses, approval fields, and `command/exec` argv plus base64 output deltas for streaming. Planner says basic interop and no real approvals, but maps command execution to AgentSession bash strings and leaves many unsupported fields as project-or-ignore. Impact: a Codex VS Code smoke can pass only by accident or fail on parameter shape, status semantics, approvalsReviewer, or command stream behavior. Fix: create a reference-pinned method matrix: accept, ignore with documented projection, reject with JSON-RPC error, or gjc-extension only. Implement `command/exec` with argv, stdout/stderr or documented PTY multiplexing, base64 deltas where streaming is claimed, and assert no approval request notifications are emitted.

6. MEDIUM — Migration cutover has the right phases but lacks hard deletion gates at public API boundaries. Evidence: `cli/args.ts` includes mode `rpc`; `main.ts` dispatches rpc/rpc-ui; `modes/index.ts` and `src/index.ts` export `runRpcMode`, `RpcClient`, and RPC types; Python and robogjc build `gjc --mode rpc`; docs still document RPC and notifications SDK. Impact: dangling SDK exports, CLI aliases, env overrides, and Python compatibility paths can survive the atomic deletion and contradict the spec no-backcompat requirement. Fix: add a cutover checklist that removes mode literals and dispatch, SDK exports, native `NotificationServer` exports after parity, docs, env/test overrides such as `GJC_HARNESS_RPC_COMMAND`, and all legacy Python protocol dataclasses once app-server suites pass.

7. MEDIUM — Schema generation feasibility is not yet proved. Evidence: current schema script is TS/Zod-only, Cargo has no schemars or ts-rs dependency, while Codex protocol generation uses both and its JSON-RPC message types are untagged and headerless. Impact: generated TS/JSON Schema can drift from runtime wire shapes, especially for tagged/untagged unions, camelCase fields, optional params, and method-specific request/notification envelopes. Fix: make the Phase 0 generator spike a blocking deliverable: add Rust protocol derives, a generator command, integration with `bun run generate-schemas` and `check:schemas`, fixtures for JSON-RPC header omission, and schema validation of sample Codex plus gjc frames.

## Recommendations
1. Amend the plan before execution with explicit contracts for TS process-global isolation, per-thread command scheduling, TSFN queue/backpressure behavior, and lifecycle event non-droppability.
2. Add a formal identity and ownership section: Rust owns immutable codex thread ids, TS owns AgentSession objects, and every callback carries generation plus backend id to prevent stale event delivery.
3. Promote event mapping from tables to a tested Rust state machine with raw gjc detail preservation and Codex item lifecycle invariants.
4. Pin a Codex reference subset in fixtures and conformance tests, especially initialize, thread/start, turn/start, turn/steer, item deltas, no-op approvals, and command/exec.
5. Treat schema generation as a Phase 0 gate, not a later fallback.
6. Add an atomic deletion checklist for TS exports, CLI mode dispatch, Python API, harness adapter, docs, native NotificationServer exports, and crate removal.

## Trade-offs
| Option | Upside | Downside | Recommendation |
|---|---|---|---|
| Rust-owned registry and protocol now | Future Rust core ready, one wire authority, schema source of truth | Exposes TS global-state and N-API concurrency risks immediately | Keep, but only with isolation and backpressure contracts before execution |
| TS-owned registry with Rust transport facade | Lower near-term risk; reuses RPC scheduler and AgentSession assumptions | Weakens native-core migration seam and leaves protocol generation outside Rust | Good antithesis, but not aligned with spec unless Rust ownership proves unsafe |
| Big-bang delete after app-server parity | Matches no-backcompat spec and avoids long-lived legacy shims | High blast radius if cutover gates miss SDK/docs/Python paths | Keep atomic delete, but add explicit gate list and reference searches |
| Codex exact parity | Best client compatibility | Too large and conflicts with GJC semantics such as no approvals | Implement a pinned v1 subset with clear reject/project/extension rules |
